# 12. Agent 设计模式：上下文 + 工具 + 循环

> 本文提炼自业界生产级 AI Agent 系统的设计实践，总结 Agent 三要素的核心设计目标和工程方法。

## 1. Agent 的本质公式

```text
Agent 能力 = 上下文质量 × 工具可用性 × 循环效率
```

**不是加和，是乘积。** 任一要素为 0，整体能力归零。

三者关系：

```text
上下文质量
  │
  │ 影响工具调用准确度
  ▼
工具可用性 ──┐
  │          │
  │          │ 共同决定
  │          │
  ▼          ▼
循环效率 ──> Agent 整体能力

相互依赖：
  上下文好 → 工具理解准 → 调用成功率高 → 循环高效
  工具可靠 → 少失败 → 上下文不被错误污染 → 质量稳定
  循环高效 → 步骤少 → 上下文积累少 → 管理更容易
```

## 2. 上下文（Context）设计

### 2.1 不可能三角

上下文设计面临三个互相冲突的目标：

```text
          精准性
          /  \
         /    \
        /  ⚠️  \
       / 不可能  \
      /  同时最优  \
     /            \
    /              \
   /________________\
经济性              效率性

三者只能取其二：
  追求精准+效率 → 成本高（大上下文）
  追求精准+经济 → 效率低（频繁裁剪重建）
  追求效率+经济 → 精准度下降（激进裁剪）
```

### 2.2 五层分区机制

核心思想：按功能和稳定性将上下文分为五个区域，不同区域采用不同的管理策略。

```text
┌─────────────────────────────────────────┐
│ Zone 1：系统区（完全固定，★★★★★）       │
│ 角色描述、工作流程、安全约束             │
│ 100% 可缓存，约 5-8K tokens             │
├─────────────────────────────────────────┤
│ Zone 2：工具区（半固定，★★★★☆）          │
│ 工具分类摘要 + 按需加载详情              │
│ 工具集不变时可缓存，初始 ~0.5K           │
├─────────────────────────────────────────┤
│ Zone 3：环境区（准静态，★★★☆☆）          │
│ OS、项目结构、Git 状态、用户规则         │
│ 会话内稳定，约 2-5K tokens               │
├─────────────────────────────────────────┤
│ Zone 4：记忆区（长期持久，★★★★☆）        │
│ 用户偏好、项目知识、历史决策             │
│ 跨会话复用，按相关性注入 1-3K tokens     │
├─────────────────────────────────────────┤
│ Zone 5：对话区（高度动态，★☆☆☆☆）        │
│ 历史对话（自动压缩）、工具调用结果        │
│ 动态 10-40K tokens                       │
└─────────────────────────────────────────┘
```

### 2.3 自动压缩机制

当对话历史积累到阈值时自动触发压缩：

```text
触发条件：
  上次请求 Token 数接近模型输入限制
  OR 总步骤数超过阈值

压缩流程：
  1. 保留最近 N 轮完整内容
  2. 压缩中间轮次为关键信息
  3. 总结早期轮次为结构化摘要
  4. 更新 Memory（提取长期有价值知识）
```

压缩摘要结构：

```text
[压缩历史]
时间范围：Rounds 1-3 (Steps 1-18)
关键操作：...
重要决策：...
技术细节：...
待解决问题：...
```

效果：Token 节省显著，关键信息不丢失，准确度影响轻微。

### 2.4 Memory 机制

通过文件系统持久化用户偏好和项目知识，实现跨会话复用。

两层架构：

```text
Layer 1：用户偏好层
  编程风格、技术栈偏好、注释语言等
  来源：自动学习 + 用户显式设置

Layer 2：项目知识层
  架构信息、关键文件、常用命令、已知问题
  来源：从对话中自动提取
```

### 2.5 稳定前缀设计

核心思想：将可缓存的固定内容放在前面，动态内容放在后面。

```text
固定部分（可缓存）：
  System Zone（角色、流程、规范）
  Tool Zone（工具集固定时）
  Environment Zone 的稳定部分

动态部分（不缓存）：
  Memory Zone（相关性动态）
  Conversation Zone（高度动态）
```

关键技巧：

```text
1. 工具列表按字母序排列
2. 不在 System Prompt 中放当前时间
3. 不在每次请求中放 requestId
4. System Prompt 版本化管理
```

## 3. 工具（Tools）设计

### 3.1 四大设计目标

```text
可调用性 ── 可靠性
    \        /
     安全性
    /        \
可扩展性 ── 性能
```

### 3.2 渐进式发现

核心思想：区分固定工具和动态工具，对动态工具按需加载。

```text
第一层：固定系统工具（完整描述）
  文件操作、代码搜索、命令执行等
  数量固定（10-20 个），Token 可控

第二层：动态扩展工具（渐进式发现）
  外部工具、领域 Skill 等
  数量可能很大（数十到上百）
  初始只提供摘要，按需加载详情
```

效果：

```text
全量加载：约 12,000 Token
分层设计：约 2,300 Token
初始节省：约 80%
```

### 3.3 增强型工具描述

核心原则：**描述清晰度 = 成功率**。

对比：

```text
❌ 简单描述：
  write_to_file: Write content to file
  path: file path
  content: file content

✓ 增强描述：
  用途说明 + 参数详解（类型/必需/格式/示例）
  + 使用示例 + 注意事项 + 安全约束
```

### 3.4 上下文感知工具

让工具执行时拥有充分的环境信息：

```text
环境信息注入：
  项目根目录
  当前工作目录
  项目文件结构
  Git 状态
  依赖信息
```

效果：路径错误率大幅降低，首次成功率显著提升。

### 3.5 智能错误反馈

错误时不只说"失败了"，还要说"为什么"和"怎么办"。

四层反馈：

```text
Level 1：错误事实（What）
  工具名称、失败状态、基本错误信息

Level 2：详细分析（Why）
  具体操作、问题诊断、可能原因

Level 3：修复建议（How）
  具体修复步骤、替代方案、示例代码

Level 4：辅助信息（Context）
  相关环境信息、类似成功案例
```

### 3.6 结果智能裁剪

保留关键信息，省略冗余细节。

裁剪规则：

```text
1. 保留开头 N 行（捕获初始输出和警告）
2. 保留结尾 N 行（捕获最终结果和状态）
3. 中间显示截断信息
4. 错误信息优先保留
5. 成功标志必须保留
6. 统计信息保留
```

## 4. 循环（Loop）设计

### 4.1 步进式执行模型

将复杂任务分解为原子化的步骤：

```text
用户输入任务
  ↓
初始化（加载上下文、准备工具）
  ↓
多步执行循环：
  Step N 开始
  ├─ 检查：是否需要裁剪上下文？
  ├─ 检查：是否达到步数限制？
  ├─ 检查：是否检测到循环？
  ├─ 调用 LLM
  ├─ 执行工具 or 返回文本
  └─ 判断是否继续
  ↓
返回结果
```

### 4.2 三层循环控制

```text
第一层：硬限制（Hard Limit）
  设置最大步数，达到时强制终止
  所有任务的兜底保护

第二层：循环检测（Loop Detection）
  记录最近 N 个操作
  检测重复模式（唯一操作数 ≤ 2 → 循环）
  警告 LLM 改变策略或提前终止

第三层：智能终止（Smart Termination）
  接近限制时提前 1-2 步注入提示
  引导 LLM 优雅总结，而非强制中断
```

### 4.3 三级 ID 追踪

```text
conversationId → 整个会话
  ├─ requestId → 单次执行
  │   ├─ messageId → Step 1
  │   ├─ messageId → Step 2
  │   └─ messageId → Step N
  └─ requestId → 下一次执行
```

### 4.4 多 Agent 协作

通过 Sub-Agent 机制实现专业化分工：

```text
主 Agent
  ├─> Sub-Agent A（独立上下文）→ 返回结果
  ├─> Sub-Agent B（独立上下文）→ 返回结果
  └─> 整合结果，继续执行

优势：
  上下文隔离（主对话保持简洁）
  专业化（定制系统提示词和工具集）
  可复用（一次定义，多处使用）
```

## 5. 系统协同与权衡

### 5.1 场景化配置

| 场景 | 上下文策略 | 工具策略 | 循环策略 |
|---|---|---|---|
| 追求准确度 | 最大化保留，保守压缩 | 全量详情，完整反馈 | 宽松限制，较多步数 |
| 追求效率 | Cache 优先，适中压缩 | 渐进发现，智能反馈 | 适中限制，智能终止 |
| 追求成本 | 激进裁剪，频繁压缩 | 按需加载，结果裁剪 | 严格限制，早期终止 |
| 生产推荐 | 分区+智能，双重阈值 | 固定+渐进，上下文感知 | 步数限制，三层防护 |

### 5.2 核心指标体系

```text
一级指标：
  任务完成率
  首次成功率
  平均完成步数
  Token 效率

二级指标：
  循环率
  Cache 命中率
  工具执行成功率
  用户干预率
```

## 6. 设计原则总结

```text
原则 1：可控性优先于智能性
  宁可让人类控制，也不要完全自主

原则 2：小步快跑优于大步慢走
  每步做一件事，问题早发现、易回滚

原则 3：优雅终止优于强制完成
  不完美的总结 > 无限循环

原则 4：状态透明优于黑盒执行
  用户应该清楚每步在做什么

原则 5：描述清晰度 = 成功率
  详细说明 + 具体示例 > 简短描述

原则 6：分层设计 = 效率
  固定工具完整 + 动态工具渐进

原则 7：上下文完整度 = 准确性
  环境信息 + 项目状态 > 孤立工具

原则 8：反馈质量 = 恢复能力
  原因分析 + 修复建议 > 简单报错

原则 9：结果裁剪 = 下一步质量
  智能保留 > 全量返回 > 简单截断
```
