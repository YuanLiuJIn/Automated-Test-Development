# 02. 自动化测试开发基础

> 目标：补齐测试开发的基础概念，理解自动化测试平台为什么要分层、为什么要工程化。

## 1. 自动化测试是什么？

自动化测试不是简单“写脚本点按钮”，而是：

```text
把人工重复验证的测试活动，转化为可重复执行、可观测、可报告、可接入流水线的工程系统。
```

它关注四件事：

```text
1. 怎么表达测试用例
2. 怎么稳定执行测试
3. 怎么判断测试是否通过
4. 怎么把结果反馈给研发流程
```

## 2. 测试金字塔

经典测试金字塔：

```text
        E2E / 系统测试         少量，慢，接近真实用户
       ─────────────────
       集成测试               中等，验证模块协作
      ───────────────────
      单元测试                大量，快，验证函数/类逻辑
```

在 UE 游戏项目中可以映射为：

| 层级 | 游戏场景 |
|---|---|
| 单元测试 | 伤害公式、技能冷却、背包数量计算、任务状态机函数 |
| 集成测试 | 任务系统 + 背包系统、UI + 角色状态、战斗 + 掉落 |
| 系统测试 | 启动游戏、进入地图、完成副本、性能回归、稳定性跑测 |

AI 测试平台通常更擅长帮助生成集成测试和系统测试，但底层单元测试仍然重要。

## 3. 测试用例的基本组成

一个标准测试用例至少包含：

```text
用例 ID
标题
优先级
适用平台
前置条件
测试步骤
预期结果
断言
超时时间
数据采集要求
清理逻辑
```

例如：

```text
用例：玩家完成副本后获得奖励
前置：玩家已接取任务，背包有空位
步骤：进入副本 → 击败 Boss → 打开奖励宝箱
预期：任务完成，背包新增奖励，结算 UI 出现
```

## 4. 断言是什么？

断言是测试通过或失败的判断标准。

```text
不是“执行完步骤就算测完”，而是必须验证结果是否正确。
```

常见断言：

```text
状态断言：QuestState == Completed
UI 断言：SettlementPanel visible
数值断言：PlayerHealth > 0
背包断言：Inventory contains RewardItem
日志断言：No Error / Crash
性能断言：avg_fps >= 45
```

## 5. 回归测试是什么？

回归测试用于确认：

```text
旧功能没有因为新改动坏掉。
```

游戏项目中常见回归：

```text
任务流程回归
战斗系统回归
背包/商城回归
UI 回归
性能回归
崩溃回归
历史 Bug 回归
```

AI 可以帮助根据历史缺陷自动生成回归用例。

## 6. 测试稳定性问题

自动化测试最常见的问题不是“不会写”，而是不稳定。

常见不稳定原因：

```text
等待条件不准确
环境初始化不干净
测试数据不固定
UI/Actor 定位不稳定
网络波动
帧率波动
异步加载未完成
用例之间互相污染
```

解决原则：

```text
时间等待 → 状态等待
硬编码定位 → 语义/ID/状态定位
人工观察 → 结构化状态回传
一次性报告 → 全过程 Trace
```

## 7. CI/CD 中的自动化测试

DevOps 流水线中，自动化测试通常分阶段：

```text
提交代码
  ↓
编译
  ↓
单元测试
  ↓
冒烟测试
  ↓
功能回归
  ↓
性能测试
  ↓
报告与质量门禁
```

不同阶段的测试目标不同：

| 阶段 | 目标 |
|---|---|
| 冒烟测试 | 快速判断构建是否基本可用 |
| 回归测试 | 检查核心功能是否被破坏 |
| 性能测试 | 检查 FPS、FrameTime、内存是否退化 |
| 稳定性测试 | 长时间运行，检查崩溃、泄漏、卡死 |

## 8. AI 在自动化测试中的位置

AI 不应该替代所有测试框架，而是应该做这些事：

```text
1. 从需求中生成测试点
2. 从测试点生成 BDD / Spec
3. 从 Spec 生成 JSON / 脚本
4. 根据失败日志做归因
5. 根据失败原因提出修复建议
6. 根据历史缺陷推荐回归用例
7. 根据代码 diff 推荐影响范围
```

AI 不适合直接无约束做：

```text
删除断言
跳过失败用例
修改业务预期
绕过核心校验
直接操作生产环境
```

## 9. 一句话总结

> 自动化测试开发的核心不是写几个脚本，而是建立一套可表达、可执行、可观测、可诊断、可持续运行的质量工程系统。AI 的价值在于提升测试设计、生成、诊断和维护效率，但底层执行和质量判断仍然需要确定性的工程框架。
